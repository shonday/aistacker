AIStacker / Overview

generate-tool

A CLI that turns a feature list into a production-ready React tool component, complete with SEO copy, registry entries, and a manual review checklist.

How it works

1 · Load environment
Reads .env.local, populates LLM API keys into process.env.
2 · Conflict check
Scans data/tools.ts to confirm the slug is not already registered. Fails immediately if it is.
3 · Build prompt → call LLM
Constructs a structured prompt with every feature, WRONG/RIGHT examples, and output format. Tries Ollama → Gemini → Anthropic → Groq in order.
4 · Parse output
Extracts === SECTION === blocks: component TSX, category, tags, SEO title, SEO description, intro, usage, example, use cases, FAQ.
5 · Static check
Validates component name, semantic Tailwind tokens, shadcn imports, no alert(), accessibility labels. Errors trigger a retry with a fix prompt.
6 · Write output
Saves ComponentName.tsx, REVIEW.md, and optionally copies the component into the project and appends registry entries automatically.
ℹ

All three retry attempts use the same LLM provider that succeeded in attempt 1. The fix prompt sends back the previous code plus the exact error list from the static checker.

AIStacker / Quick Start

Get running in 3 steps

From zero to your first generated tool.

Step 1 — Set up your API key

Pick one provider. Gemini Flash is recommended: free tier is generous and fast.

# in .env.local
GEMINI_API_KEY=your-key-here
# get it at: aistudio.google.com/apikey

Step 2 — Run your first tool

npx tsx scripts/generate-tool.ts \
  --id       "jwt-decoder" \
  --name     "JWT Decoder" \
  --desc     "Decode and inspect JWT tokens — header, payload, expiry — in your browser." \
  --features "decode header as JSON,decode payload as JSON,show expiry status,copy individual sections"

Output after ~20 seconds:

✅  Generated in 18.4s via gemini
    Component : ./output/JwtDecoder.tsx
    Review    : ./output/REVIEW.md

Step 3 — Review and deploy

# Open the checklist
cat output/REVIEW.md

# Start dev server and verify
npm run dev
# → open http://localhost:3000/tools/jwt-decoder

# If it looks good, copy to project manually OR re-run with --project-dir
npx tsx scripts/generate-tool.ts --id "jwt-decoder" ... --project-dir .
✓

Tip: Always run without --project-dir first. Review the output, then re-run with --project-dir . to auto-copy and auto-register. This gives you a chance to check before it touches your registry files.

AIStacker / All Parameters

CLI Parameters

Complete reference for all generate-tool.ts flags.

FlagTypeDescription
--id REQUIRED Kebab-case slug. Becomes the URL (/tools/jwt-decoder), the filename (JwtDecoder.tsx), and the registry key. Must be unique.
--name REQUIRED Human-readable display name shown in the UI and page title. Example: "JWT Decoder"
--desc REQUIRED One-sentence description used in cards and meta tags. Keep it under 160 chars. Be specific — this directly affects SEO.
--features REQUIRED Comma-separated list of features the LLM must implement. The more specific, the better. Each feature becomes a checkbox in REVIEW.md.
--category optional One of: formatter encoder generator tester converter japanese text number color image network crypto. Auto-detected by the LLM if omitted.
--tags optional Comma-separated tags for search and cross-linking. Auto-generated by LLM if omitted. Example: "jwt,token,decoder,base64,auth"
--provider optional auto (default) | ollama | gemini | anthropic | groq. auto tries providers in priority order until one succeeds.
--attempts optional Max retry attempts when static checks fail. Default: 3. Each retry sends the errors back to the LLM with a fix prompt.
--output optional Directory for generated files. Default: ./output. Safe to run multiple times — files are overwritten.
--project-dir optional If set, automatically copies the component to {project-dir}/components/tools/ and appends entries to data/tools.ts and lib/toolRegistry.ts.
--batch optional Path to a JSON file containing an array of request objects. All other flags are ignored when --batch is set.
AIStacker / Single Tool

Single Tool Generation

Generating and deploying one tool at a time.

Generate only (safe, no project changes)

npx tsx scripts/generate-tool.ts \
  --id       "color-converter" \
  --name     "Color Converter" \
  --desc     "Convert colors between HEX, RGB, HSL, and HSB with a live preview." \
  --features "hex input,rgb input,hsl output,hsb output,live color swatch,copy any format" \
  --category "color"

Generate and auto-deploy to project

npx tsx scripts/generate-tool.ts \
  --id       "color-converter" \
  --name     "Color Converter" \
  --desc     "Convert colors between HEX, RGB, HSL, and HSB with a live preview." \
  --features "hex input,rgb input,hsl output,hsb output,live color swatch,copy any format" \
  --project-dir .

# What --project-dir does automatically:
#   1. Copies ColorConverter.tsx → components/tools/ColorConverter.tsx
#   2. Appends entry  → data/tools.ts
#   3. Appends line   → lib/toolRegistry.ts

Force a specific provider

# Use Anthropic instead of auto-fallback
npx tsx scripts/generate-tool.ts \
  --id "color-converter" ... \
  --provider anthropic

# Use local Ollama (offline, private)
npx tsx scripts/generate-tool.ts \
  --id "color-converter" ... \
  --provider ollama

Increase retries for complex tools

# Allow up to 5 retry attempts (useful for tools with many features)
npx tsx scripts/generate-tool.ts \
  --id "markdown-preview" ... \
  --attempts 5
AIStacker / Batch Mode

Batch Generation

Generate multiple tools from a JSON manifest. Tools are processed sequentially with a 2-second pause between each to respect rate limits.

Batch JSON format

Create a file at scripts/codegen/batch/wave1.json:

[
  {
    "toolId":      "jwt-decoder",
    "displayName": "JWT Decoder",
    "description": "Decode and inspect JWT tokens instantly.",
    "features": [
      "decode header as JSON",
      "decode payload as JSON",
      "show expiry status",
      "copy individual sections"
    ]
  },
  {
    "toolId":      "color-converter",
    "displayName": "Color Converter",
    "description": "Convert between HEX, RGB, HSL, and HSB.",
    "features": [
      "hex input", "rgb input", "hsl output", "live color swatch"
    ],
    "category": "color"
  }
]

Run the batch

npx tsx scripts/generate-tool.ts --batch scripts/codegen/batch/wave1.json

# With auto-deploy to project
npx tsx scripts/generate-tool.ts \
  --batch       scripts/codegen/batch/wave1.json \
  --project-dir .

Output summary at the end:

[batch] Done — 5 succeeded, 1 failed.
⚠

Batch + --project-dir: Registry files are appended after each successful tool. If a later tool fails, the earlier ones are already in the registry. Check git diff before committing.

Pre-configured wave 1

The file scripts/codegen/batch/wave1.json already contains 5 tools ready to generate: JWT Decoder, Japanese Word Counter, Color Converter, Hash Generator, and Markdown Preview.

AIStacker / Testing a Tool

How to Test a Generated Tool

The generator produces a REVIEW.md with a specific checklist. Here's how to work through it systematically.

The REVIEW.md checklist

Every generated tool produces a REVIEW.md in ./output/. Open it first.

cat output/REVIEW.md

It contains: the feature checklist, static check results (errors and warnings), and the registry snippets to paste.

Manual testing steps

  • ✓
    Start dev server
    npm run dev → open http://localhost:3000/tools/{tool-id}
  • ✓
    Test every feature in REVIEW.md
    Each feature you specified in --features is a numbered checkbox. Verify each one works.
  • ✓
    Empty input
    Clear all inputs and trigger the action. Tool must show a graceful error state, not crash or show a blank screen.
  • ✓
    Invalid input
    For parsers (JSON, JWT, Regex): enter garbage. Expect an inline error message, not a thrown exception.
  • ✓
    Copy buttons
    Every copy button must visually confirm success (e.g., "Copied ✓" for 1.5s). No alert() popups.
  • ✓
    Dark mode
    Add class="dark" to <html> in DevTools. All text must remain readable. No white boxes on dark backgrounds.
  • ⚠
    Mobile layout (375px)
    Resize browser to 375px. Buttons must not overflow. Two-column layouts must stack.
  • ⚠
    Large input
    Paste 10,000 characters. The tool must not freeze the browser tab (check for O(n²) re-renders).
  • ✓
    SEO content below the tool
    Scroll down. The intro, usage, example, use cases, and FAQ sections must all render with correct formatting.

Checking static check warnings

REVIEW.md lists any warnings from the static checker. These are not blocking errors but should be fixed before shipping:

## Static check result
Errors: 0
Warnings: 2

- [WARN] hardcoded-color (line 34): Hardcoded Tailwind color found.
- [WARN] a11y-input-label (line 61): <input> has no id or aria-label.

Fix these manually in the generated file or re-run with a stronger --features description.

What passes vs fails automatically

✓ component name matches ✓ "use client" directive ✓ shadcn imports via @/components/ui ✓ no alert() calls ⚠ hardcoded Tailwind colors ⚠ inputs without labels ⚠ icon buttons without aria-label ⚠ useEffect empty deps array
AIStacker / Upgrading a Tool

How to Upgrade an Existing Tool

The generator detects slug conflicts and refuses to overwrite. Upgrading requires a deliberate workflow.

⚠

Running generate-tool with an existing --id will fail immediately with registry_conflict. This is intentional — upgrades require the steps below.

Option A — Regenerate and replace manually

Use this when you want full control over what changes.

# 1. Generate to ./output (does NOT touch the project)
npx tsx scripts/generate-tool.ts \
  --id       "regex-tester" \
  --name     "Regex Tester" \
  --desc     "Test regex with real-time highlighting, flag toggles, and group capture breakdown." \
  --features "pattern input,flag toggles g i m s,live match highlighting,group capture panel,match count,copy pattern"

# 2. Review output/RegexTester.tsx vs existing component
diff output/RegexTester.tsx components/tools/RegexTester.tsx

# 3. If satisfied, replace
cp output/RegexTester.tsx components/tools/RegexTester.tsx

# 4. Update data/tools.ts manually if description/tags changed
#    (the registry entry is in output/REVIEW.md for reference)

Option B — Rename slug to force regeneration

Use this when you want a completely fresh registry entry (e.g., the old tool had a poor SEO description).

# 1. Delete old entry from data/tools.ts and lib/toolRegistry.ts
# 2. Delete old component file
rm components/tools/RegexTester.tsx

# 3. Now generate-tool will accept the slug again
npx tsx scripts/generate-tool.ts \
  --id "regex-tester" ... \
  --project-dir .

Targeted feature upgrade (manual)

When you only need to add one feature to an existing tool, editing directly is faster than regenerating:

# Open the component
code components/tools/RegexTester.tsx

# Ask your editor AI or Claude:
# "Add a match count display below the test string textarea.
#  Show '3 matches' in muted text. Use the existing regex state.
#  Follow the shadcn token conventions already in this file."
✓

Rule of thumb: Use the generator for new tools and large rewrites. For adding 1–2 features to an existing tool, edit the component directly — it's faster and safer than regenerating.

AIStacker / Static Checker

Static Checker Rules

The checker runs automatically after every LLM response. Errors block output and trigger a fix prompt. Warnings are recorded in REVIEW.md but don't block.

RuleSeverityWhat it catches
use-client error File does not start with "use client". All tool components are client components.
component-name error The export default function name doesn't match the PascalCase of --id. Prevents import failures in toolRegistry.
shadcn-import error Importing directly from radix-ui or @radix-ui/react-* instead of @/components/ui/*.
no-alert error Use of alert(). Must be replaced with inline copied-state feedback.
hardcoded-color warn Tailwind palette classes like text-gray-500, bg-slate-100. Must use semantic tokens instead.
a11y-button-label warn Icon-only button without aria-label.
a11y-input-label warn <input> without id or aria-label.
hooks-effect-deps warn useEffect with empty [] dependency array — flags potential stale closure bugs.

Reading the error output

# In the terminal when a check fails:
[gen] Static check failed (2 errors):
component-name: Component is named "MyTool" but must be "JwtDecoder".
no-alert: alert() is not allowed. Use inline copied state. (line 47)

# In REVIEW.md:
Errors: 0
Warnings: 1
- [WARN] hardcoded-color (line 34): Use semantic tokens.
AIStacker / Writing Good Prompts

Writing Good Feature Lists

The --features flag is the single biggest lever on output quality. The description and features together determine 90% of what the LLM builds.

The core rule

Features are instructions to the LLM, not user stories. Write them as specific UI behaviors, not abstract goals.

❌ Too vague

--features "encode and decode,copy,clear"

✓ Specific and actionable

--features "encode text to base64 on button click,decode base64 to text on button click,show error for invalid base64 input,copy result to clipboard with 1.5s copied feedback,clear both fields button,live character count on input"

Feature anatomy

A good feature entry has three parts: what UI element, what trigger, and what result.

PatternExample
input + action "paste JSON and press Format button to see prettified output"
live/reactive "live match count updates as user types the pattern"
error state "show red inline error when JSON is invalid, with line number"
copy behavior "copy result button shows 'Copied ✓' for 1.5s then resets"
mode toggle "toggle between encode and decode modes, preserves input"
output format "show all four hashes simultaneously in a grid, not a dropdown"

Description quality matters for SEO

The --desc is used verbatim in the tool card and meta description. Write it for a search engine result snippet.

❌ Generic

--desc "A tool that converts colors."

✓ Specific + keyword-rich

--desc "Convert colors between HEX, RGB, HSL, and HSB formats with a live color swatch preview."

Feature count vs quality

ℹ

Sweet spot: 5–8 features. Under 4 features → the LLM adds filler. Over 12 → the component becomes too large to generate reliably in one pass and static checks start failing. For complex tools, split them: one generation per major mode.

AIStacker / LLM → Prompt

Use an LLM to Write Your Prompts

Yes — using an LLM to expand your natural language idea into a precise generate-tool command is the right approach. Here's exactly how.

The meta-prompt

Copy this prompt and use it with Claude, ChatGPT, or Gemini. Replace the last line with your tool idea.

You are helping me prepare a command for a tool generator called generate-tool.
The generator accepts these parameters:

--id       kebab-case slug (URL-safe, unique)
--name     Display name
--desc     One-sentence SEO description, max 160 chars, specific and keyword-rich
--features Comma-separated list of specific UI behaviors the tool must implement
--category One of: formatter encoder generator tester converter japanese text number color image network crypto

Rules for --features:
- Each feature describes a specific UI behavior: what element, what trigger, what result
- Include error states ("show inline error when X is invalid")
- Include copy behavior ("copy button shows 'Copied ✓' for 1.5s")
- Include live/reactive behavior if appropriate ("updates as user types")
- 5-8 features is the sweet spot
- No vague items like "good UX" or "easy to use"

My tool idea: [YOUR IDEA IN PLAIN LANGUAGE]

Output ONLY the generate-tool command, ready to run. No explanation.

Example: from idea to command

Your input to the LLM

My tool idea: I want a tool that lets users convert between
different number bases — like decimal, hex, binary, and octal.
Should be easy to use and show all conversions at once.

LLM output

npx tsx scripts/generate-tool.ts \
  --id       "number-base-converter" \
  --name     "Number Base Converter" \
  --desc     "Convert numbers between decimal, binary, octal, and hexadecimal bases instantly." \
  --features "decimal input field that updates all other bases live,binary output (base 2),octal output (base 8),hexadecimal output (base 16) with uppercase letters,show error for non-numeric input,copy individual output with 1.5s feedback,clear all button" \
  --category "converter"

Why this works well

The meta-prompt forces the LLM to translate abstract ideas ("easy to use", "show all conversions") into concrete UI specifications that generate-tool can act on. You get the convenience of natural language with the precision of a spec.

✓

Best workflow: Describe your idea to an LLM → get the command → run it → check REVIEW.md → fix any warnings → commit. The whole cycle typically takes under 5 minutes per tool.

Iterating on a bad result

If the generated tool is missing something or got a feature wrong, feed the REVIEW.md back to the LLM:

Here is the REVIEW.md from a generated tool:
[paste REVIEW.md]

The tool is missing the group capture panel feature.
Give me an updated --features list that adds it clearly,
and keep all the existing features.
